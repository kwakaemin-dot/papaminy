## 🚀 Chaos Engineering Phase 1 시나리오 A 실행 계획 (Failover & Retry)

**목표:** 시스템 안정성 목표($S_M=99.9\%$) 달성을 위해 Failover 및 Retry 메커니즘의 신뢰성을 실증하고, 잠재적 장애 상황에서의 복구 능력을 검증한다.

### 1. 인프라 요구사항 (Infrastructure Requirements)

시스템 안정성 확보를 위한 핵심 지표($S_{LER/PL}$)와 $99.9\%$ Uptime 목표 달성을 위해 다음의 기술 스택 및 아키텍처 변경이 필수적입니다.

| 영역 | 요구사항 상세 내용 | 근거/목표 |
| :--- | :--- | :--- |
| **서비스 아키텍처** | **Active-Passive 또는 Active-Active 구성:** 핵심 서비스(API Gateway, DB 연결 모듈)는 최소 2개 이상의 독립적인 리전 또는 데이터센터에 분산 배포되어야 한다. | $S_{LER/PL}$ 확보 및 재해 복구 능력 극대화 |
| **Failover 메커니즘** | **자동 스위칭 로직:** 서비스 간의 상태를 실시간으로 모니터링하여, 특정 노드 또는 리전 장애 감지 시 트래픽을 자동으로 백업 시스템으로 전환하는 로직(Health Check 기반)이 필수적이다. | Failover 기능의 신뢰성 확보 |
| **Retry 메커니즘** | **지능형 재시도 (Intelligent Retry):** 외부 API 호출이나 DB 접근 실패 시, 단순 재시도가 아닌 지수 백오프(Exponential Backoff) 전략과 최대 재시도 횟수를 정의해야 한다. | 서비스의 안정적 복구 및 부하 분산 |
| **데이터 계층** | **분산 트랜잭션 관리:** 데이터 일관성을 보장하기 위해 SAGA 패턴 또는 2PC/3PC와 같은 분산 트랜잭션 관리 기법을 도입하여 Failover 시 데이터 정합성 문제를 방지해야 한다. | 데이터 무결성 유지 |
| **모니터링 통합** | **$S_{LER/PL}$ 실시간 측정:** Prometheus를 통해 시스템의 지연 시간(Latency), 에러율(Error Rate), 처리량(Throughput)을 1초 단위로 수집하고, 이를 Grafana 대시보드에 시각화해야 한다. | $S_M=99.9\%$ 목표의 실시간 검증 근거 확보 |
| **캐싱 레이어** | **Redis/Memcached 분산 캐싱:** 외부 API 호출 및 데이터 조회 시 DB 부하를 줄이고 Latency 병목을 해소하기 위해 모든 읽기 요청에 대해 분산 캐싱 계층을 의무화한다. | $P95 Latency < 500ms$ 목표 달성 지원 |

### 2. 최소한의 테스트 환경 구성안 (Minimum Test Environment Setup)

Chaos Engineering 시나리오를 안전하게 실행하기 위해 실제 프로덕션 환경을 모사하는 격리된 테스트 환경이 필요합니다.

1.  **테스트 환경 분리:**
    *   **Staging/Pre-Production 환경 구축:** 프로덕션 데이터의 복제본을 사용하여, 실제 서비스에 영향을 주지 않고 Chaos 실험을 수행할 수 있는 독립적인 환경을 구성한다. (이는 $S_{LER/PL}$ 측정의 기반이 됨)
2.  **테스트 시나리오 정의 및 자동화:**
    *   **시나리오 A 집중:** Failover(노드 장애 주입)와 Retry(네트워크 지연 또는 서비스 응답 지연 주입)에 초점을 맞춘 테스트 스크립트를 개발한다.
    *   **자동화 도구 사용:** Chaos Mesh, Gremlin 등 외부 도구를 활용하여 정의된 시나리오를 자동 실행하고, 시스템의 복구 시간(RTO)과 데이터 손실률(RPO)을 자동으로 측정하는 래퍼 스크립트를 작성한다.
3.  **측정 지표 설정:**
    *   **기준선 설정:** Chaos 실험 전, 정상 상태에서의 $S_{LER/PL}$ (Latency, Error Rate) 기준치를 명확히 기록한다.
    *   **복구 측정:** Failover 및 Retry가 발생했을 때 시스템이 목표 Latency($<500ms$)로 복귀하는 데 걸리는 시간(Recovery Time)과 그 과정에서 발생한 에러 로그를 상세하게 기록하도록 모니터링 파이프라인을 구성한다.
