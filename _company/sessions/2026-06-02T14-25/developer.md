### ⚙️ 시스템 아키텍처 설계 (API/Worker 분리 및 모니터링)

데이터 흐름의 예측 가능성을 확보하기 위해 **Producer-Consumer 패턴**을 기반으로 API 서버와 워커 프로세스를 명확히 분리하고, 상태를 중앙에서 모니터링하는 구조를 제안합니다.

#### 1. 아키텍처 개요
*   **API 서버 (Producer):** 외부 요청을 받아 메시지 큐에 작업을 발행(Publish)하고, 작업의 상태를 조회하는 역할만 수행합니다. 데이터 처리 로직은 실행하지 않습니다.
*   **메시지 큐 (Broker):** API와 워커 간의 비동기 통신 및 작업 대기열 역할을 합니다. (예: Redis/RabbitMQ 등)
*   **워커 프로세스 (Consumer):** 메시지 큐에서 작업을 가져와 실제 DFRS 데이터 처리 및 계산을 수행합니다.
*   **헬스 체크 모듈:** 각 컴포넌트(API, MQ, Worker)의 상태를 주기적으로 보고하여 전체 시스템의 예측 가능성을 확보합니다.

#### 2. 데이터 흐름 상세 설계 (DFRS 기반)

1.  **요청 발생:** 클라이언트가 `GET /api/kpi/...` 요청을 API 서버에 보냅니다.
2.  **메시지 발행:** API 서버는 해당 KPI 계산 요청을 **메시지 큐(Queue)**에 작업 메시지로 발행합니다 (예: `KPI_CALC_REQUEST`). 이 시점에서 API 응답은 즉시 완료됩니다.
3.  **워커 처리:** 워커 프로세스는 큐에서 `KPI_CALC_REQUEST`를 수신하고, DFRS 데이터베이스에서 필요한 데이터를 조회하여 KPI를 계산합니다.
4.  **상태 업데이트:** 워커는 작업이 성공했는지 실패했는지의 상태를 **상태 저장소(Status Store)**에 기록합니다 (예: `KPI_CALC_SUCCESS` 또는 `KPI_CALC_FAILED`).
5.  **모니터링:** API 서버나 별도의 모니터링 서비스는 상태 저장소를 주기적으로 읽어 전체 데이터 흐름의 지연 여부 및 에러 발생률을 감시합니다.

#### 3. 기술 스택 제안 (구현 관점)
*   **API 서버:** FastAPI/Node.js (요청 수신 및 큐 발행 담당)
*   **워커 프로세스:** Python Worker (메시지 큐 리스너 및 실제 계산 담당)
*   **메시지 큐:** Redis (Pub/Sub 또는 Stream) 혹은 RabbitMQ (더 견고한 트랜잭션 보장 시)
*   **상태 모니터링:** 간단한 인메모리 딕셔너리 또는 별도의 DB 테이블에 작업 ID와 상태를 저장.
