### 1. Chaos Engineering 시나리오 정의 (Targeting $S_{LER/PL}$)

시스템의 핵심 제약 조건(Failover/Retry, Latency Bottleneck Mitigation)을 검증하기 위해 다음 세 가지 주요 시나리오를 정의합니다.

#### 시나리오 A: 비동기 실패 및 복구 테스트 (Failover & Retry Mechanism Verification)
*   **목표:** 외부 API 연동 모듈의 비동기 실패 시 안전한 복구 로직(Failover/Retry mechanism)이 정상 작동하는지 검증합니다.
*   **시나리오:** 핵심 데이터 요청 중 50% 확률로 외부 서비스(예: 외부 API 또는 DB 레플리카)에 타임아웃을 발생시켜, 시스템이 즉시 안전하게 다른 경로로 전환하고 재시도하는지 확인합니다.
*   **측정 지표:** 복구 시간(Recovery Time), DLQ 이동 성공률, 최종 사용자 경험 지표.

#### 시나리오 B: Latency 병목 및 캐싱 검증 (Latency Bottleneck Mitigation Verification)
*   **목표:** 데이터베이스 쿼리 지연 및 외부 API 호출의 병목 구간에서 Redis 캐싱 레이어가 P95 Latency 목표($<500ms$)를 달성하는 데 기여하는지 검증합니다.
*   **시나리오:** 특정 트래픽 부하(Load Test) 시나리오를 실행하여, 캐시 히트율이 예상치에 미달할 경우 DB 접근을 허용하고 캐시에서 데이터를 가져오는 로직의 지연 시간 변화를 측정합니다.
*   **측정 지표:** P95 Latency (캐시 사용 시 vs. DB 직접 접근 시), 캐시 적중률(Hit Rate).

#### 시나리오 C: 시스템 부하 및 지속성 테스트 (Uptime & Load Verification)
*   **목표:** 시스템이 예상치 못한 최대 부하 조건에서도 99.9% Uptime 목표를 유지하는지 확인합니다.
*   **시나리오:** 동시 사용자 요청 수를 평소 대비 $2\times$ 이상 증가시킨 상태에서, 비동기 처리 중 일부 요청에 대해 의도적인 지연(Latency Injection)을 발생시켜 시스템의 안정적 처리를 검증합니다.
*   **측정 지표:** 시스템 다운타임, 오류율(Error Rate), 서비스 가용성($S_{LER/PL}$) 실시간 모니터링.
